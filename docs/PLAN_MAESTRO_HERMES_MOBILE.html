1. Visión General

Hermes Mobile evolucionó de una app read-only (vendedor solo ve datos) a una plataforma comercial completa donde el vendedor genera datos en campo, usa un asistente IA con voz, y gerencia consume todo desde el desktop.

VENDEDOR (campo, celular) GERENCIA (oficina, desktop) ┌──────────────────────┐ ┌──────────────────────┐ │ Hermes Mobile v3.5 │ │ Hermes Desktop │ │ │──── escribe ────► │ │ │ • Scorecard │ │ • Reportes │ │ • Clientes + saldos │ │ • Dashboards │ │ • Ventas │ │ • Mapas │ │ • Pedidos │ Supabase │ • Overview CRM │ │ • Visitas GPS │ (cloud) │ • Config objetivos │ │ • Agenda │ │ • Edición prompts │ │ • Cronos (IA + voz) │◄──── lee ──────── │ de agentes IA │ │ • Google Calendar │ │ │ └──────────────────────┘ └──────────────────────┘ │ │ └─────── SQL Server (VPN, read-only) ───────┘ (facturación + pedidos + CxC)
Pilares técnicos: Flutter (Dart) + Supabase (PostgreSQL cloud) + SQL Server (jTDS via VPN) + OpenAI GPT-4o-mini (Cronos) + Whisper API (voz) + Google Calendar API + flutter_local_notifications + local_auth + flutter_secure_storage + geolocator.

Implicancia para rollout: vendedores con v3.7.2 o anterior tienen que desinstalar y reinstalar v3.7.3+ una sola vez (Android rechaza signature mismatch).

3.4 Arquitectura de datos offline (v3.5)

Patrón "offline-first" para la cartera de clientes:

  • ClientesCache persiste en SharedPreferences con timestamp
  • ClientesService.getClientes() intenta SQL Server con timeout; si falla usa cache; si no hay cache throw
  • UI muestra "Actualizada hace X" y banner amarillo "Sin VPN" cuando corresponde
  • Efecto: Cronos, agenda, visitas, registrar actividades funcionan sin VPN. Solo el scorecard y ventas requieren SQL Server en vivo.

Patrón complementario de broadcaster (DataEvents): notifica cambios a todas las pantallas sin acoplarlas.

9. Extra: Cronos — Asistente IA con Voz ✨ No planeado

Feature estrella no planeada

Nació como POC y terminó siendo una de las funciones más usadas. Es un asistente de agenda conversacional con voz, restringido al dominio comercial, que entiende lenguaje natural argentino.

Capacidades actuales (v3.4)

  • Agendar actividades — llamada, visita, propuesta, reunión, recordatorio
  • Cargar visita con GPS — detecta "estoy visitando a X" / "llegué a Y" y toma ubicación
  • Consultar pendientes — filtros hoy/mañana/semana/mes/vencidas/fecha_especifica + por cliente + orden próxima/última + límite
  • Consultar visitas hechas — por período o por cliente
  • Completar tareas — "ya llamé a García"
  • Múltiples acciones en un solo mensaje (incluidas recurrencias)
  • Charla casual — saludos, gracias, perfecto → respuesta amable
  • Desambiguación de cliente — si hay varios homónimos, chips para elegir
  • Ayuda contextual — "¿qué podés hacer?" → lista capacidades

Stack técnico de Cronos

  • LLM: OpenAI GPT-4o-mini (~$0.001 por conversación)
  • Voz: Whisper API (español argentino, $0.006/min)
  • Grabación: paquete record v5 (aac, 16kHz)
  • System prompt: en Supabase (agent_prompts), editable sin recompilar
  • Historial: conversación completa enviada al LLM (multi-turn)
  • Placeholders en el prompt: {{vendedor}}, {{fecha}}, {{clientes}}

UX visual

  • Pantalla de bienvenida con chips rotando (4 visibles, 15 en pool, rotan cada 4s)
  • Ficha de Cronos — tap en nombre → bottom sheet con misión/capacidades/ejemplos
  • Overlay de grabación con pulso reactivo al audio + timer + botones Listo/Cancelar
  • Burbuja "Transcribiendo..." animada mientras Whisper procesa
  • Burbuja "Pensando..." con 3 puntos bouncing mientras LLM responde
  • Tarjetas de acción con entrance animado + check elástico al confirmar
  • Card de visita GPS: chips de motivo tocables + GPS en vivo + botón deshabilitado hasta tener ubicación
  • Haptic feedback en todas las interacciones críticas

11. Extra: Google Calendar (Bloque L) ✨ No planeado

Sincronización bidireccional con Google Calendar. Toda actividad con fecha puede aparecer en el calendario del vendedor.

Modos de sincronización

  • Desactivado (default)
  • Manual — botón "Agregar al calendario" en cada actividad
  • Automático — toda actividad nueva con fecha va al calendar

Comportamiento

  • Crear actividad → insert event en calendar
  • Editar fecha/descripción → patch event
  • Eliminar actividad → delete event
  • Columna google_event_id en actividades_cliente linkea los dos lados
  • Token OAuth cifrado en flutter_secure_storage
  • Paquetes: google_sign_in, googleapis, googleapis_auth, extension_google_sign_in_as_googleapis_auth
Setup pendiente del usuario: crear OAuth client ID en Google Cloud Console con SHA-1 del keystore, habilitar Calendar API, agregar test users (los gmails de los vendedores). Cuando esté listo, el toggle "Conectar con Google" empieza a funcionar. Código ya preparado para recibirlo.

12. Extra: System Prompts en DB ✨ No planeado

Arquitectura para agregar agentes IA sin tocar código. Todos los prompts viven en Supabase y se pueden editar en caliente.

-- Tabla
agent_prompts (
  id SERIAL PRIMARY KEY,
  agent_id TEXT UNIQUE,     -- 'cronos', 'analista', 'cobrador', etc
  prompt TEXT,
  description TEXT,
  version INT DEFAULT 1,
  active BOOLEAN,
  updated_at TIMESTAMP,
  updated_by TEXT
)

PromptService — patrón

  • Cache local 5 minutos → cero overhead por llamada
  • Fallback hardcodeado si la DB falla (resiliencia offline)
  • Placeholders {{var}} reemplazados en runtime
  • Método invalidate(agent_id) para forzar recarga

Futuro: pantalla en Hermes Desktop para editar prompts de todos los agentes con versionado y rollback.

20. Cronos Analytics — Reportes y Conclusiones

Contexto: en v3.6.0 instrumentamos cada turno de Cronos en la tabla cronos_logs. La instrumentación fue barata (fire-and-forget desde la app, no afecta latencia percibida) pero abre una superficie de análisis muy rica. Esta sección documenta qué reportes podemos sacar, qué decisiones podemos tomar con cada uno, cómo extraerlos, y qué columnas faltarían para reportes más profundos.

20.1 Datos disponibles hoy

Tabla cronos_logs:

ColumnaTipoPara qué sirve
idBIGSERIALPK autoincremental
vendedor_nombreTEXTQuién hizo el turno
user_msgTEXTLo que dijo o tipeó el vendedor
response_rawTEXTJSON crudo devuelto por el modelo (o body del response si falló HTTP)
response_mensajeTEXTTexto que vio el vendedor en pantalla
acciones_countINTCuántas tarjetas se generaron (0 = el modelo respondió pero no propuso nada)

20.2 Categorías de análisis

Las queries se pueden correr directamente en el SQL editor de Supabase hoy mismo. Las agrupamos en 5 categorías por tipo de decisión que habilitan.

A. Salud operacional Crítico — diario

¿Cronos está respondiendo bien? ¿Hay regresiones tras cambios de prompt o de OpenAI?

A.1 Parse failures (el LLM rompió el JSON)

SELECT user_msg, response_raw, latencia_ms, created_at
FROM cronos_logs
WHERE parse_ok = FALSE AND modelo != 'bypass'
ORDER BY created_at DESC LIMIT 20;

Conclusión: Si aparecen frases que repetidamente rompen el JSON, hay que ajustar el system prompt. Si suben tras un cambio de prompt → revertir. Si OpenAI devuelve errores HTTP, lo vemos acá.

A.2 Latencia P50 / P90 por modelo

SELECT modelo,
       COUNT(*) AS turnos,
       PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY latencia_ms) AS p50,
       PERCENTILE_DISC(0.9) WITHIN GROUP (ORDER BY latencia_ms) AS p90,
       MAX(latencia_ms) AS max_ms
FROM cronos_logs
WHERE created_at > NOW() - INTERVAL '7 days'
GROUP BY modelo;

Conclusión: Confirma el ahorro de latencia con bypass (~200ms vs ~1500ms). Si el P90 del LLM se va por encima de 3s persistente → degradación de OpenAI; cambiar el modelo o reducir contexto.

A.3 % de fallas por día

SELECT DATE(created_at) AS dia,
       COUNT(*) FILTER (WHERE parse_ok = FALSE) AS fails,
       COUNT(*) AS total,
       ROUND(100.0 * COUNT(*) FILTER (WHERE parse_ok = FALSE) / COUNT(*), 2) AS fail_pct
FROM cronos_logs
WHERE modelo != 'bypass' AND created_at > NOW() - INTERVAL '30 days'
GROUP BY DATE(created_at) ORDER BY 1;

Conclusión: Tendencia diaria. Meta: <2% de fails. Spikes señalan problema concreto a investigar.

B. Costo y eficiencia Mensual

¿Cuánto estamos gastando? ¿Las quick actions están ahorrando lo que esperábamos?

B.1 Ratio bypass vs LLM

SELECT modelo, COUNT(*) AS turnos,
       ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 2) AS pct
FROM cronos_logs
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY modelo;

Conclusión: Validás que las quick actions están siendo usadas. Meta: que bypass >= 40% del tráfico. Si es bajo, los chips no son lo bastante útiles o el vendedor no los descubre — repensar layout o agregar nuevas quick actions.

B.2 Costo estimado en USD

-- Estimación: ~2K tokens-in + ~500 tokens-out promedio por turno LLM
-- gpt-4o-mini: $0.15/M tokens-in + $0.60/M tokens-out
SELECT
  COUNT(*) FILTER (WHERE modelo != 'bypass') AS turnos_llm,
  ROUND(COUNT(*) FILTER (WHERE modelo != 'bypass')
        * (2000 * 0.15 + 500 * 0.60) / 1000000.0, 2) AS usd_estimado_30d
FROM cronos_logs
WHERE created_at > NOW() - INTERVAL '30 days';

Conclusión: Permite proyectar a más vendedores. Si vamos a escalar de 10 a 30 vendedores, multiplicar por 3 da el costo esperado. Es el dato de input para la decisión "vale la pena el proxy?" (deuda técnica #1).

C. Calidad del prompt y comprensión del modelo Quincenal

¿Cronos entiende lo que le piden? ¿Hay tipos de pedido que el modelo no resuelve?

C.1 Mensajes que NO generaron acciones (modelo desorientado)

SELECT user_msg, response_mensaje, COUNT(*) AS veces
FROM cronos_logs
WHERE modelo != 'bypass'
  AND acciones_count = 0
  AND parse_ok = TRUE
  AND created_at > NOW() - INTERVAL '30 days'
GROUP BY user_msg, response_mensaje
ORDER BY veces DESC LIMIT 30;

Conclusión: Frases reales donde Cronos respondió pero no hizo nada concreto. Es la fuente más rica para iterar el prompt: cada fila es una expectativa del vendedor que el modelo no cumplió. Decidir caso por caso si:

  • Es algo legítimo que el prompt debería soportar → agregar al prompt
  • Es algo fuera del scope → reforzar el rechazo amable en el prompt
  • Es un caso muy frecuente → considerar agregar una quick action

C.2 Frecuencia de "no te puedo ayudar"

SELECT DATE(created_at), COUNT(*) AS rejects
FROM cronos_logs
WHERE response_mensaje ILIKE '%no puedo%'
   OR response_mensaje ILIKE '%solo puedo%'
   OR response_mensaje ILIKE '%no soy capaz%'
GROUP BY 1 ORDER BY 1 DESC LIMIT 30;

Conclusión: Si esto sube, el prompt es muy restrictivo o los vendedores intentan usar Cronos para cosas no soportadas (lo que abre oportunidades de producto: tal vez quieren cargar pedidos por voz, hacer cobranzas, etc).

D. Adopción por vendedor Mensual

¿Quién usa la herramienta? ¿Quién no? ¿Por qué?

D.1 Ranking de uso (últimos 30 días)

SELECT vendedor_nombre,
       COUNT(*) AS turnos,
       COUNT(*) FILTER (WHERE modelo = 'bypass')  AS quick_actions,
       COUNT(*) FILTER (WHERE modelo != 'bypass') AS llm,
       MAX(created_at) AS ultimo_uso
FROM cronos_logs
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY vendedor_nombre
ORDER BY turnos DESC;

Conclusión: Detecta:

  • Power users (10+ turnos/día) — entrevistarlos para entender el caso de uso real
  • Vendedores que no la abren — capacitación, resistencia al cambio o el caso de uso no aplica
  • Vendedores que abandonaron (último uso > 14 días) — algo los frustró; retroalimentar

D.2 Curva de aprendizaje semanal

SELECT vendedor_nombre,
       DATE_TRUNC('week', created_at) AS semana,
       COUNT(*) AS turnos
FROM cronos_logs
GROUP BY 1, 2 ORDER BY vendedor_nombre, semana;

Conclusión: Si el uso de un vendedor crece semana a semana, la herramienta "cuajó". Si se estanca o cae después del primer pico, hay un problema de UX o de fit.

E. Decisiones de producto Trimestral

¿Qué tendríamos que construir/eliminar/mejorar basándonos en datos?

E.1 Top frases que pegan al LLM (candidatas a quick action)

SELECT LOWER(user_msg) AS frase, COUNT(*) AS veces
FROM cronos_logs
WHERE modelo != 'bypass'
  AND created_at > NOW() - INTERVAL '60 days'
GROUP BY LOWER(user_msg)
HAVING COUNT(*) >= 5
ORDER BY veces DESC LIMIT 20;

Conclusión: Si "qué tengo este mes" aparece 30 veces y siempre va al LLM, cuesta tokens y latencia. Convertirla en quick action ahorra dinero y mejora UX.

E.2 Quick actions más usadas

SELECT user_msg, COUNT(*) AS uses
FROM cronos_logs
WHERE modelo = 'bypass'
  AND created_at > NOW() - INTERVAL '30 days'
GROUP BY user_msg
ORDER BY uses DESC;

Conclusión: Si una quick action no se usa nunca, sacarla y ganar espacio para otra. Si todas tienen uso similar, mantener.

E.3 Horarios pico de uso

SELECT EXTRACT(hour FROM created_at AT TIME ZONE 'America/Argentina/Buenos_Aires') AS hora,
       COUNT(*) AS turnos
FROM cronos_logs
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY hora ORDER BY hora;

Conclusión: Confirma uso real en horario laboral (descarta testing/fraude). Identifica franjas críticas — si todos usan Cronos entre 9 y 11 AM, ese es el momento donde no se puede tener downtime de OpenAI.

20.3 Limitaciones actuales y columnas a sumar

Hoy no podemos responder estas preguntas con la tabla actual:

Pregunta de negocioPor qué no se puede hoyPara arreglarlo

Estas mejoras son ortogonales y se pueden sumar de a una. Cada una abre nuevas preguntas. El orden recomendado por valor:

  1. tokens_in + tokens_out (costo exacto, no estimado)
  2. input_mode (validar que la voz funciona, no solo que existe)
  3. confirmed (qué % de las propuestas el vendedor las acepta — proxy de utilidad)
  4. intent_top + cliente_codigo_target (analítica fina)
  5. session_id (análisis de conversaciones completas, no turnos sueltos)

21. Force Update — Killswitch remoto de versión

Contexto: hasta v3.6 los updates eran 100% opcionales — el vendedor veía un banner amarillo en Configuración y decidía cuándo actualizar. Para fixes críticos (ej: bug de cobranza, security fix, breaking change en una query) hace falta poder obligar a que todos pasen a la versión nueva. v3.7 introduce este mecanismo sin recompilar la app.

21.1 Cómo funciona

Tabla nueva app_config en Supabase con un row clave:

CREATE TABLE app_config (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

INSERT INTO app_config (key, value)
VALUES ('min_version_required', '3.0.0');

Flujo cada vez que un vendedor hace login:

  1. La app lee app_config.min_version_required desde Supabase.
  2. Compara con su PackageInfo.version local.
  3. Si local < min_required → muestra ForceUpdateScreen bloqueante (un solo botón: "Actualizar ahora") y guarda el último valor leído en SharedPreferences para soportar offline.
  4. Si local >= min_required → continúa al HomeScreen normalmente y dispara pre-download del APK en background si hay release nuevo en GitHub.

21.2 Cómo forzar un update a todos los vendedores

Una sola query en Supabase, sin tocar el código:

UPDATE app_config
SET value = '3.7.1', updated_at = NOW()
WHERE key = 'min_version_required';

A partir de ahí, cualquier vendedor con versión < 3.7.1 que abra la app queda bloqueado en la pantalla de update hasta que actualice. Sin release nuevo, sin recompilar.

Para volver a permisivo (después de que todos actualizaron):

UPDATE app_config SET value = '3.0.0' WHERE key = 'min_version_required';

21.3 Edge cases manejados

EscenarioComportamiento
Vendedor sin red al hacer login (sin VPN, sin datos)Si la app ya leyó el valor antes (cache local), lo usa. Si nunca leyó, no bloquea (no tenemos info para decidir).
Falla la descarga del APKPantalla bloqueante con botón "Reintentar" + opción "Cerrar sesión" (escape de emergencia).
Vendedor cierra la app forzadamenteAl volver a abrir, el gate vuelve a aparecer. No hay forma de saltearlo.
Flag mal seteado (typo de versión, bloquea por error)Reversible al instante desde Supabase. Próximo login → cae el bloqueo.
Back de Android en la pantalla bloqueantePopScope(canPop: false) — bloquea el back. El único escape es "Cerrar sesión".

21.4 Pre-download del APK (mejora UX para todos los updates)

Independiente del force update: al hacer login, si hay un release nuevo (cualquier release, no solo crítico), la app descarga el APK silencioso en background. Cuando el vendedor toca "Actualizar" en Configuración, el instalador abre instantáneo en lugar de esperar el minuto de descarga.

  • Cache: getTemporaryDirectory()/hermes_<tag>.apk
  • Tag y path guardados en SharedPreferences para validar coherencia
  • Si el APK ya está pre-descargado para ese tag, no se vuelve a descargar (idempotente)
  • Si la pre-descarga falla (sin red, etc), no se notifica al vendedor — es transparente; el flujo manual sigue funcionando

21.5 Niveles de update

NivelTriggerUICuándo usarlo
Soft (existía desde v1.3)Hay release más nuevo en GitHubBanner amarillo en Configuración + tile "Actualizar"Features nuevas, mejoras no críticas

Coexisten: el soft sigue funcionando como antes. El force se suma como killswitch para casos donde no hay otra opción.

21.6 Limitación importante

Android todavía requiere 1 tap del vendedor en el dialog del instalador del sistema (no se puede automatizar sin Google Play o MDM, según research de abril 2026). Lo que conseguimos con force update es:

  • El vendedor no puede usar la app hasta actualizar (antes podía postergar indefinidamente)
  • El APK ya está descargado por el pre-download (1 tap → instalador → confirmar → reabrir)
  • Total: ~30 segundos desde que ve el bloqueo hasta volver a usar la app

Si en algún momento crece el equipo (>20 vendedores) y vale la pena, podemos evaluar Managed Google Play o MDM para conseguir update 100% silencioso, pero hoy el costo/beneficio no justifica.

22. Auditoría de Seguridad y Hardening

Contexto: en mayo 2026 se hizo una auditoría completa de seguridad de la app contra OWASP Mobile Top 10 (MASVS). El threat model considerado es: distribución privada, instalación manual a un equipo de ~10 vendedores, app no publicada en Play Store. El riesgo principal es un vendedor descontento que se va con el APK en el teléfono, no un atacante externo sofisticado.

22.1 Hallazgos resueltos en v3.7.3

MED-3 No había network_security_config.xml explícito M5 — Insecure Communication Archivo creado y referenciado en manifest. Solo CAs del sistema (rechaza user-installed certs → previene MITM con cert empresarial o malware).

22.2 Hallazgos descartados (no aplican al threat model)

Hallazgos que el audit identificó pero que se decidió no atacar dada la naturaleza privada de la distribución:

MED-2 Cleanup de cronos_logs (retención) Datos en Supabase cloud, solo el admin tiene acceso. Los vendedores no ven la DB. Logs son valiosos para iterar el prompt y analytics (§20). No hay obligación regulatoria para borrarlos.
MED-5 MainActivity exportada Necesario para el launcher. No hay deep links → riesgo teórico.

22.3 Hallazgos pendientes — sprint v3.8.0

🛡️ CRIT-2 + CRIT-3 — Proxy de OpenAI + DB credentials Alta

Problema unificado: tanto la API key de OpenAI (AppConfig.openaiApiKey) como las credenciales de Supabase Postgres (AppConfig.pgPass) viven en lib/config/constants.dart y se compilan al APK. Ambas son extraíbles con strings app-release.apk.

Mitigación actual:

  • Cap mensual de $400 USD en OpenAI Dashboard (compartido con otros proyectos del workspace)
  • Release keystore (CRIT-1 ya resuelto) → solo APKs firmados por nosotros se aceptan, lo que limita impersonación
  • Force update + min_version_required → si se filtra una APK, podemos rotar credenciales y empujar update

Solución definitiva: Supabase Edge Functions como proxy. La key/credenciales viven server-side; la app autentica con un token por vendedor. Plan completo documentado en docs/PLAN_PROXY_OPENAI.md:

  • 3 endpoints: auth-token, cronos-chat, cronos-transcribe
  • Auth: token por vendedor (Opción 2 del plan, recomendada)
  • Tabla uso_llm para tracking granular per-vendedor
  • Rate limit per-vendedor (100 chat/h, 60 transcribe/h)
  • Esfuerzo: 1 día completo (8 fases secuenciales documentadas)
  • Costo: $0 (Supabase Free cubre 500k invocaciones/mes; uso esperado ~11k/mes)
