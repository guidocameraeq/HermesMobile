1. Visión General

Hermes Mobile evoluciona de una app read-only (vendedor solo ve datos) a un CRM bidireccional donde el vendedor genera datos en campo y gerencia los consume desde el desktop.

5. Bloque E — Actividades de Cliente (Historia Clínica)

Prioridad ALTA — Base de todo lo demás

Objetivo: Que cada cliente tenga un "historial clínico" completo con todas las interacciones que el vendedor tuvo con él: visitas, llamadas, propuestas comerciales, presentaciones de producto, reuniones. El vendedor carga actividades desde la ficha del cliente y el historial se construye automáticamente.
Situación actual

Hoy la ficha del cliente en Hermes Mobile muestra datos transaccionales (facturación, saldo CxC, evolución mensual, líneas de producto) pero no tiene registro de las interacciones comerciales. El vendedor no puede anotar que llamó a un cliente, que le presentó un producto, o que le envió una propuesta. Esta información se pierde o queda en la cabeza del vendedor.

Las visitas GPS ya se registran (Bloque A), pero son solo un check-in geolocalizado — no hay forma de registrar qué se habló ni qué resultó de la visita.

Situación deseada

Cada cliente tiene un timeline cronológico (historia clínica) en su ficha que muestra, de más reciente a más antiguo:

  • Visitas GPS registradas (ya existen en tabla visitas)
  • Actividades cargadas manualmente (llamadas, propuestas, presentaciones)
  • Relevamientos completados (matriz potencial, PDV — de bloques F y G)
  • Cada entrada con: icono por tipo, descripción, fecha/hora

El vendedor puede abrir la ficha de cualquier cliente y ver rápidamente: "hace 2 semanas lo visité, le presenté la línea de pinturas, quedó en evaluar, hoy tengo que llamarlo". El jefe puede ver esta misma información desde Hermes Desktop para evaluar la gestión comercial del vendedor.

Por qué es la base de todo

Este bloque es el primero porque establece la estructura de datos que todos los demás bloques alimentan. La matriz de potencial (F), el relevamiento PDV (G), la agenda (I) — todos generan registros que aparecen en esta historia clínica. Sin esta base, los demás bloques quedan aislados.

  • Botón "Cargar actividad": Bottom sheet con dropdown de tipo + campo descripción + botón guardar.

6. Bloque C — Embudo CRM de Prospectos

Prioridad ALTA — Reemplazar CRM actual

Objetivo: Reemplazar el CRM actual (Armin) con un embudo de prospectos nativo en Hermes Mobile. Los vendedores crean prospectos desde la app con un formulario custom (reemplaza Google Forms), los mueven por un embudo de 6 etapas, registran actividades, y marcan como ganado/perdido con motivo.
Situación actual — Lo que no funciona

El equipo comercial usa un CRM externo (Armin) que fue adaptado con parches para funcionar con el modelo de prospección de la empresa. Los problemas principales:

  • Alta de prospectos: Se hace via Google Forms porque el CRM no tiene un formulario sencillo. El vendedor completa un formulario largo de Google, eso va a un Google Sheet, y con Zapier se carga al CRM como "nuevo cliente" (que en realidad es un prospecto). Es engorroso y propenso a errores.
  • Embudo: Se adaptó la herramienta de embudo del CRM a las etapas del negocio, pero no fue diseñado para esto. Las tarjetas se mueven entre etapas manualmente.
  • Actividades: El CRM tiene un módulo de actividades pero es "muy malo, no se entiende, difícil de usar" y no tiene notificaciones ni se sincroniza con el calendario.
  • Tracking limitado: Cuando un prospecto se "gana", el alta del cliente se hace por fuera (formulario Excel a administración). No hay forma de vincular un prospecto ganado con el nuevo cliente en el sistema ERP.
  • Overview: Un chico se encarga de hacer el seguimiento de la cartera de prospectos, saca reportes, y empuja a los vendedores. Hoy hace esto desde el CRM que queremos reemplazar.
Situación deseada

Todo el circuito de prospección vive en Hermes:

  • Formulario nativo en la app: El vendedor crea prospectos directamente desde Hermes Mobile con un formulario custom adaptado al negocio. Sin Google Forms intermediario.
  • Embudo visual: Lista de prospectos filtrable por etapa con resumen de cantidad y valor por etapa. El vendedor mueve prospectos entre etapas con un tap.
  • 6 etapas: Entrada → 1er Contacto/Entrevista → Propuesta Enviada → Seguimiento → Análisis Financiero → Prospecto Estratégico. Más Ganado/Perdido como resultado final.
  • "Prospecto Estratégico" es una etapa especial para prospectos grandes a largo plazo donde las negociaciones pueden tardar meses o años.
  • Actividades por prospecto: Timeline de interacciones (llamadas, emails, reuniones, propuestas).
  • Ganado/Perdido: Se registra el resultado con motivo. El tracking llega hasta ahí — el alta del cliente sigue siendo manual por fuera por ahora.
  • Admin/Overview: Se hace desde Hermes Desktop. El chico de overview ve el embudo completo de todos los vendedores desde la PC.
Etapas fijas ahora, configurables después

Las etapas se guardan en una tabla de Supabase (etapas_embudo) con seed inicial de las 6 etapas actuales. Por ahora son fijas — si hay que cambiar una, se edita directamente en la base de datos. En una fase posterior se agrega UI de configuración en Hermes Desktop para que el admin pueda crear, editar y reordenar etapas sin tocar código.

7. Bloque D — Leads desde Google Forms

Prioridad MEDIA — Depende del Bloque C

Circuito: Cliente potencial completa Google Form → Zapier → Supabase tabla leads → Admin califica en Hermes.exe → Asigna a vendedor → Vendedor recibe notificación → Completa formulario largo → Se crea prospecto en embudo.
Situación actual

Previo a los prospectos hay dos embudos de leads:

  • Embudo de Talleres: Contactos de talleres pequeños que no son clientes directos. El chico de overview los revisa y los deriva por WhatsApp al distribuidor que corresponda según la zona.
  • Embudo de Leads: Leads calificados que el chico lee, evalúa y asigna al vendedor según la zona geográfica.

Los formularios de leads son Google Forms porque los completa gente externa (clientes potenciales que llegan por la web, redes sociales, etc.). Este formulario debe seguir siendo externo y "compartible" — no puede estar dentro de la app porque no es para vendedores sino para el público.

Hoy el flujo es: Google Form → Google Sheet → Zapier → CRM (Armin). Queremos cambiarlo a: Google Form → Google Sheet → Zapier → Supabase.

Situación deseada
  • El Google Form sigue funcionando igual para el público externo
  • Zapier/Make envía los datos a una tabla leads en Supabase via REST API
  • El admin (chico de overview) ve los leads desde Hermes Desktop — los califica como Lead o Taller, y los asigna a vendedores
  • Cuando un lead se asigna a un vendedor, este recibe una notificación en Hermes Mobile
  • El vendedor va a visitar/llamar al lead y cuando está listo, completa el formulario largo de prospecto directamente en la app → el lead se convierte en prospecto y entra al embudo (Bloque C)

Tabla Supabase

CREATE TABLE leads (
    id SERIAL PRIMARY KEY,
    nombre TEXT NOT NULL,
    email TEXT,
    telefono TEXT,
    empresa TEXT,
    rubro TEXT,
    origen TEXT DEFAULT 'google_forms',
    tipo TEXT DEFAULT 'lead',  -- 'lead' | 'taller'
    estado TEXT DEFAULT 'nuevo', -- 'nuevo' | 'asignado' | 'descartado'
    vendedor_asignado TEXT,     -- lo pone el admin
    notas_admin TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

Configuración Zapier/Make

  • Trigger: New Row in Google Sheets
  • Action: POST a https://<project>.supabase.co/rest/v1/leads
  • Headers: apikey, Authorization, Content-Type, Prefer
  • Body: mapeo de columnas del form a campos de la tabla

8. Bloque F — Matriz de Potencial

Prioridad ALTA — Depende del Bloque E

Objetivo: El vendedor completa un cuestionario por cliente que, mediante fórmulas de ponderación, calcula el valor monetario potencial de ese cliente para la empresa. Se hace periódicamente (cada 3 meses). El resultado se muestra en la ficha del cliente y alimenta un reporte de cartera.
Situación actual

No existe un relevamiento de potencial en la empresa. Los vendedores tienen una noción intuitiva de cuánto "podría comprar" cada cliente, pero no hay un método estructurado para medirlo. La gerencia no tiene visibilidad del potencial real de la cartera.

Situación deseada

El vendedor abre la ficha de un cliente → toca "Relevamiento de potencial" → se abre un cuestionario con preguntas específicas del negocio (tamaño del local, rubros que maneja, zona, competencia, etc.). Cada respuesta tiene un peso numérico. Al finalizar, el sistema calcula automáticamente un valor monetario potencial ($) que ese cliente podría representar para la empresa.

  • Contexto al cargar: Mientras completa el cuestionario, ve datos clave del cliente — facturación últimos 6 meses, líneas que compra, última visita. Esto ayuda a responder con información real.
  • Control de periodicidad: No puede hacer dos veces el mismo relevamiento en el mismo trimestre (UNIQUE constraint). A los 3 meses del último relevamiento, la app muestra un badge/alerta: "Hace 3 meses que no relevás a este cliente".
  • Resultado visible: El valor de potencial aparece en la ficha del cliente y en la historia clínica.
  • Reporte de cartera: En un apartado separado, el vendedor puede ver un ranking de su cartera por potencial. La gerencia ve lo mismo desde Hermes Desktop.
  • Preguntas configurables: Las preguntas y pesos se definen en la tabla config_cuestionarios, que el admin gestiona desde el desktop. Si cambian los criterios, no hay que tocar código.

Tabla Supabase

CREATE TABLE matriz_potencial (
    id SERIAL PRIMARY KEY,
    vendedor_nombre TEXT NOT NULL,
    cliente_codigo TEXT NOT NULL,
    cliente_nombre TEXT NOT NULL,
    respuestas JSONB NOT NULL,       -- {pregunta_id: respuesta}
    ponderacion_total NUMERIC,        -- valor $ calculado
    mes INTEGER, anio INTEGER,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(vendedor_nombre, cliente_codigo, mes, anio)
);

CREATE TABLE config_cuestionarios (
    id SERIAL PRIMARY KEY,
    tipo TEXT NOT NULL,  -- 'matriz_potencial' | 'relevamiento_pdv'
    nombre TEXT NOT NULL,
    preguntas JSONB NOT NULL,  -- [{pregunta, tipo_respuesta, opciones, peso}]
    activo BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

Controles

  • Duplicado: UNIQUE constraint impide cargar dos veces el mismo período
  • Alerta 3 meses: Al abrir la ficha del cliente, verificar si la última matriz tiene más de 3 meses → mostrar badge/alerta
  • Contexto: Al cargar, mostrar facturación últimos 6 meses del cliente

9. Bloque G — Relevamiento PDV

Prioridad MEDIA — Depende del Bloque E

Objetivo: El vendedor puede hacer un relevamiento del punto de venta del cliente: completar un cuestionario sobre exhibición, competencia, stock visible, y tomar 1-2 fotos del local/góndola. Todo queda guardado y accesible desde la ficha del cliente y desde Hermes Desktop para gerencia.
Situación actual

No hay un mecanismo para documentar el punto de venta. Los vendedores a veces sacan fotos con el celular personal pero no quedan en ningún sistema centralizado. Gerencia no tiene visibilidad de cómo se exhiben los productos en cada cliente.

Situación deseada
  • El vendedor abre la ficha del cliente → toca "Relevamiento PDV" → cuestionario con preguntas sobre el punto de venta
  • Puede tomar 1-2 fotos con la cámara del celular
  • Las fotos se suben a Supabase Storage (bucket de imágenes, no a la tabla) y se guardan las URLs
  • El relevamiento queda en la historia clínica del cliente
  • Desde Hermes Desktop, gerencia puede ver los relevamientos con fotos — útil para evaluar exhibición, hacer seguimiento de materiales de marketing, etc.
  • Las preguntas son configurables desde la tabla config_cuestionarios (misma infraestructura que la matriz de potencial)

Fotos

10. Bloque H — Objetivo de Visitas Diarias

Prioridad MEDIA — Independiente

Objetivo: Establecer una meta diaria de visitas por vendedor y comparar automáticamente con las visitas GPS registradas. El vendedor ve su progreso y gerencia ve el cumplimiento del equipo.
Situación actual

Las visitas GPS ya se registran (Bloque A completado), pero no hay un objetivo contra el cual compararlas. El vendedor sabe que "fue a 3 clientes hoy" pero no sabe si eso es suficiente. Gerencia no tiene una métrica de actividad comercial diaria.

Situación deseada
  • Desde Hermes Desktop, el admin configura una meta de visitas diarias por vendedor (ej: "LAMBRECHT: 5 visitas/día")
  • En la app, el vendedor ve un badge: "3/5 visitas hoy" — en la pestaña Más y/o en el Scorecard
  • La tabla visitas ya tiene todos los datos necesarios — solo falta la meta y la comparación
  • En Hermes Desktop, gerencia ve un reporte: cumplimiento de visitas por vendedor por semana/mes
  • Eventualmente este objetivo puede integrarse como una métrica más del Scorecard (como facturación o aperturas)

11. Bloque I — Agenda + Notificaciones

Prioridad MEDIA — Depende del Bloque E

Situación actual

Los vendedores no tienen un sistema de agenda integrado con la gestión comercial. Usan su calendario personal (Google Calendar, notas del celular) para agendar llamadas y visitas, pero eso no se conecta con el historial del cliente. Cuando el vendedor dice "tengo que llamar a Juan la semana que viene", esa intención no queda registrada en ningún sistema.

Situación deseada

El vendedor tiene una "Mi agenda" dentro de Hermes con actividades programadas. Cuando crea una, puede asignarle una fecha y hora y el celular le manda una notificación local a la hora indicada (funciona con la app cerrada, sin servidor, sin costo). Al tocar la notificación se abre Hermes, marca la actividad como completada, y esa acción queda registrada en la historia clínica del cliente (Bloque E).

Opcionalmente, cada actividad tiene un botón "Agregar a Google Calendar" para sincronizar con el calendario personal del vendedor (se ve en el reloj, en la PC, etc.).

Además, el sistema genera alertas automáticas: si hace más de 3 meses que no se relevó la matriz de potencial de un cliente, aparece un badge en la ficha y opcionalmente una notificación.

12. Bloque J — Voz + LLM + Push (Fase Avanzada)

Prioridad BAJA — Fase futura, depende de E + I

Visión

El dueño de la empresa quiere que eventualmente el vendedor pueda cargar actividades con un mensaje de voz: "Hoy llamé a García de Distribuidora Norte, le presenté la línea de pinturas vintage, quedó en evaluar la propuesta y me va a confirmar la semana que viene". Un LLM (inteligencia artificial) procesa esa transcripción y genera automáticamente la actividad con tipo, cliente, descripción y fecha de seguimiento.

Por otro lado, se necesitan push notifications desde el servidor para que gerencia pueda enviar mensajes al equipo ("reunión mañana a las 9") y para que el sistema envíe alertas automáticas ("te asignaron un nuevo lead", "tenés 3 seguimientos vencidos").

Grabación de voz → Actividad automática

  1. Vendedor graba mensaje de voz (30 seg - 2 min)
  2. Transcripción: speech_to_text local (gratis) o Whisper API (~$0.006/min)
  3. Texto → Claude API → extrae: tipo actividad, cliente, resumen, próximo paso
  4. Se guarda como actividad con origen = 'voz'
  5. El vendedor puede revisar/editar antes de confirmar

Push Notifications (Firebase Cloud Messaging)

  • Gerencia manda notificaciones a vendedores desde Hermes.exe
  • Sistema manda alertas automáticas (leads asignados, seguimientos vencidos)
  • Requiere proyecto Firebase + configuración Android
  • Es complementario a las notificaciones locales del Bloque I (que no necesitan servidor)
Nota: Este bloque es ambicioso pero cada pieza es independiente. Se puede hacer primero las push notifications (Firebase) y después la voz (LLM), o al revés. La voz requiere costos recurrentes (API de transcripción + API de LLM) que hay que evaluar según el volumen de uso.
